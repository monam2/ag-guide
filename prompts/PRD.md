# PRD

- 제품 요구 사항 문서. 일반적으로 PM(Project Manager)가 작성한다.
- 개발팀이 무엇을 만들어야하는지, 요구사항 명세가 정의되어 있다.

## 작성법

### 1. 명확한 목표를 설정해야 한다.

- 무엇을 만들지 한 문장으로 정의
- 성공을 측정할 수 있는 구체적 지표를 제시
- ex. 회원가입/로그인 기능을 만든다면 이메일 인증 완료율이나 비밀번호 복구 성공률 같은 수치 목표를 명시

```
## 목표
사용자가 이메일과 비밀번호로 회원가입하고 로그인할 수 있다.

### 성공 기준
- 회원가입 후 이메일 인증 완료율 80% 이상
- 로그인 실패 시 명확한 오류 메시지 제공
- 비밀번호 찾기 기능으로 24시간 내 90% 복구
```

### 2. 구체적인 요구사항을 작성한다.

- 각 기능에 고유한 식별자를 부여
- 입력 필드는 무엇인지, 어떤 검증 규칙을 적용할지, 예외 상황은 어떻게 처리할지 등을 상세히 기술한다.
- ex. 회원가입이라면 이메일 형식 검증, 비밀번호 규칙, 중복 체크 같은 세부 사항을 모두 담아야 한다.

```
## 기능 요구 사항
...

### FR-001: 회원가입
- 입력 필드: 이메일, 비밀번호, 비밀번호 확인, 닉네임
- 이메일 형식 검증 (정규식)
- 비밀번호 규칙: 8자 이상, 영문+숫자+특수문자
- 중복 이메일 체크
- 가입 완료 시 인증 이메일 발송

### FR-002: 로그인
- 입력 필드: 이메일, 비밀번호
- 로그인 실패시 오류 메시지(이메일 없음 / 비밀번호 틀림 구분 안함)
- 5회 실패 시 15분 잠금
- "로그인 유지" 체크박스 (7일)
```

### 3. 기술 명세에는 실제 구현에 필요한 기술적 결정사항을 명시한다.

- 인증 방식은 무엇을 쓸지, 토큰 만료 시간은 어떻게 설정할지, API 엔드포인트는 어떤 구조로 만들지 정리한다.

```
## 기술 요구 사항

### 인증 방식
- JWT(Access Token + Refresh Token)
- Access Token: 15분
- Refresh Token: 7일
- HttpOnly, Secure 쿠키 사용

### API 엔드포인트 경로

| 메서드 | 경로 | 설명 |
|---|---|---|
| POST | /api/auth/register | 회원가입 |
| POST | /api/auth/login | 로그인 |
| POST | /api/auth/logout | 로그아웃 |
| POST | /api/auth/refresh | 토큰 갱신 |
```

### 4. 제약사항과 범위를 명확히 구분해야 한다.

- 포함할 기능과 제외할 기능을 분명히 나누어 작성한다.
- 범위 확대를 방지하고 팀의 집중력을 유지할 수 있다.
- 소셜 로그인이나 2단계 인증 같은 기능은 다음 단계로 미루는 것도 적절하다.

```
## 범위

### 포함 (In Scope)
- 이메일/비밀번호 인증
- 비밀번호 찾기/재설정
- 기본 프로필 (닉네임)

### 제외 (Out of Scope)
- 소셜 로그인(Phase2로 이관)
- 2단계 인증(Phase3로 이관)
- 프로필 이미지
```

만약 PRD 작성이 어렵다면 머릿속에 떠오르는 내용을 나열한 후 AI에게 PRD를 작성해달라고 요청할 수 있다.

```
사용자가 회원가입과 로그인을 할 수 있는 사이트를 만들고 싶어. 다음 사항을 고려해서 PRD를 작성해줘.
- 이메일/비밀번호 인증 필수
- 구글 로그인 필수
- 카카오 로그인 필수
- 한 페이지로 구성
```
