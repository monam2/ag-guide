Optimizing React Rendering Performance: A Deep Dive

Alex Johnson

Oct 24, 2023 • 5 min read

React's rendering behavior is fundamental to understanding how to build performant applications. In this post, we will explore common pitfalls with re-renders, how `useMemo` actually works under the hood, and when you should avoid premature optimization.

The Problem with Re-renders

Every time a state change occurs in a parent component, React recursively re-renders all children by default. While React is fast, doing unnecessary work can lead to dropped frames in complex UIs.

Consider the following scenario where a simple counter update causes an expensive list to re-calculate:

SlowComponent.jsx 
import React, { useState } from 'react';
function Dashboard() {
  const [count, setCount] = useState(0);
  // This runs on every render!
  const expensiveData = calculateExpensiveData();
  return (
    <div>
      <h1>Count: {count}</h1>
      <button onClick={() => setCount(c => c + 1)}>
        Increment
      </button>
      <ExpensiveList data={expensiveData} />
    </div>
  );
}

To fix this, we need to ensure referential stability for props passed to child components. This is where useMemo comes in handy.

Visualizing the component tree updates

When to optimize?

  • When interactions feel sluggish (input lag).
  • When rendering large lists or tables.
  • When profiling shows specific components re-rendering too often.
"Premature optimization is the root of all evil." — Donald Knuth

Start by building your feature correctly. Use the React DevTools Profiler to identify bottlenecks before sprinkling `memo` everywhere.
